# "Кодэ"

> Usually, by changing the environment, people mean going on vacation, but I prefer to change the compiler.
> 
> `./tcc -B./runtime checker.c -o checker` → `./checker`

## First look

The zip gives you a whole tcc toolchain (`tcc` binary + a `runtime/` dir with headers and `libtcc1.a`) plus one tiny source file:

```c
extern int audit(const char *answer);

int main(void) {
    char answer[128];
    puts("Build provenance check");
    fputs("receipt: ", stdout);
    if (!fgets(answer, sizeof answer, stdin)) return 1;
    answer[strcspn(answer, "\n")] = 0;
    puts(audit(answer) ? "accepted" : "rejected");
    return 0;
}
```

`audit()` is never defined anywhere in `checker.c`, and it's not in `libtcc1.a` either (checked with `nm`/`ar t`). So where does it come from? You're told to compile with _this specific_ `tcc`, which is the giveaway — the compiler itself is the payload.

## The compiler is haunted

`strings tcc | grep audit` turns up the entire source of `audit()` sitting inside the `tcc` binary as a literal string — this custom build (`0.9.28rc ... mob@...`) silently injects that function into anything it compiles. No object file needed, no linker flag, it's baked into the compiler. Cute.

The embedded source is basically:

- a 512-byte `vm_blob` (encrypted bytecode) baked in as constant data,
- a `relocation_key()` function that opens `/proc/self/exe`, walks its own ELF `.rela` sections, and folds all the relocation entries into a 64-bit key,
- `audit()`, which uses that key to XOR-decrypt `vm_blob` into an mmap'd RWX page, then runs it as a tiny per-character bytecode VM against your input.

The comment in the source even calls out the `mprotect(code, 512, PROT_READ|PROT_EXEC)` line as _"the intended dynamic-analysis breakpoint"_ — thanks, that's basically a solution hint.

The neat trick here: the decryption key depends on the **relocations of the final compiled `checker` binary itself**, not on anything in `tcc`. So you can't just extract `vm_blob` and decrypt it in isolation — you have to actually build `checker` with the provided toolchain first, because the key is a function of how _that specific binary_ got linked. Since the compiler/linker are fixed and deterministic, building it always gives the same VM bytecode.

## Getting the bytecode out

So, do what the challenge says:

```
./tcc -B./runtime checker.c -o checker
```

Then instead of guessing, just break exactly where the source told us to:

```
gdb -batch -x cmds --args ./checker
```

with a breakpoint on `mprotect` conditioned on `rdx==5 && rsi==512` (i.e. `PROT_READ|PROT_EXEC`, 512 bytes — filters out the dynamic loader's own relro `mprotect` calls). At that point the code buffer is already decrypted in memory, sitting at whatever address is in `rdi`. Dump it:

```
dump binary memory codebuf.bin $rdi ($rdi+512)
```

## The VM

Bytecode layout, straight from the leaked source:

```
code[0..1] = n           (number of "instructions" == expected input length)
code[2]    = 0x51        (magic sanity byte)
then n instructions, 7 bytes each:
  0xA7                    opcode marker
  k1                      byte added to input char before mixing into state
  r                       rotate amount (mod 32)
  want (4 bytes, LE)      expected state value after this step
```

The state update per character is:

```c
state ^= input[i] + k1;
state  = rotl32(state, r);
state += 0x9e3779b9 ^ (i * 0x45d9f3b);
// must now equal `want`
```

Every step here is invertible (XOR, rotate, add are all reversible), and `state` only depends on the _previous_ state plus known constants from the bytecode. So there's no need to brute force anything — just walk the instructions in order and solve backwards for `input[i]` at each step:

```python
pre_add  = (want - add) & MASK
pre_rot  = rotr32(pre_add, r)
input[i] = (state ^ pre_rot) - k1   # mod 256
state    = want   # move on to next char
```

Ran that over the 51 dumped instructions and out popped:

```
grodno{Fabrice_Bellard_is_a_really_cool_programmer}
```

Fed it back into the real `./checker`:

```
$ echo "grodno{Fabrice_Bellard_is_a_really_cool_programmer}" | ./checker
Build provenance check
receipt: accepted
```

`accepted`. Nice.

(Fabrice Bellard being the actual original author of TCC is a fun touch — the compiler is quite literally vouching for its own creator.)

## tl;dr

1. Weird custom `tcc` secretly links an `audit()` backdoor into everything it compiles.
2. That function decrypts a tiny bytecode VM using a key derived from the compiled binary's own ELF relocations — so you must actually build `checker` with the given toolchain, you can't shortcut it from `tcc` alone.
3. Compile → gdb breakpoint on the `mprotect(PROT_READ|PROT_EXEC)` call → dump the decrypted 512-byte VM buffer.
4. The per-character state machine is fully invertible, so solve for the input directly instead of brute-forcing 51 unknown bytes.
5. Flag: `grodno{Fabrice_Bellard_is_a_really_cool_programmer}`
